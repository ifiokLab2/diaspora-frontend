Project Documentation: E-Commerce & Listings Platform

1. Project Overview

Diaspora Black is a high-performance, hybrid marketplace that integrates traditional Retail E-Commerce (physical products) with a Service Listings engine (professional services). The platform is built with a decoupled architecture to ensure high scalability, SEO optimization, and a seamless multi-tenant experience for both Sellers and Customers…

  • Retail E-Commerce (physical goods)
  • Service Listings Marketplace (professional services)

1. Shop Management: The Digital Identity

Section Details
ShopProfile The ShopProfile serves as the merchant’s brand anchor. Unlike a standard user profile, the ShopProfile contains sensitive commercial and branding data.
Professional Branding Sellers can upload high-resolution logos and banners. The system utilizes Cloudinary’s transformation API to automatically resize these assets for different UI components (e.g., circular thumbnails in search results vs. wide banners on shop pages).
Business Credentials Includes fields for Business Registration Numbers (RC/TIN), ensuring compliance with local marketplace regulations.
Banking & Settlement Logic Sellers must provide a Sort Code and Account Number. This data is encrypted at rest and is used by the backend to automate the routing of funds during the payout cycle.
Location SEO Shops are indexed by Country and City, allowing the “Shops Near Me” feature to function via targeted PostgreSQL queries.

2. Product & Service Management (Listing CRUD)

Section Details
Overview The platform distinguishes between Physical Products (inventory-based) and Professional Services (time or project-based).
Multi-Image Gallery Support Sellers can upload multiple images per listing. The backend handles this via a ProductImage model with a ForeignKey relationship to the main Product. This allows for a dedicated “Gallery Slider” on the frontend.
Unique Slug Architecture To ensure SEO-friendly URLs, every listing generates a unique slug. If a seller lists “White T-Shirt” and another seller does the same, the system appends a short UUID (e.g., /products/white-t-shirt-8f2b) to prevent routing conflicts.
Pricing Models Physical: Supports original_price and discount_price, automatically calculating the “Percentage Off” badge. Services: Supports Fixed, Hourly, and Negotiable pricing structures to accommodate different professional industries.

3. Analytics Dashboard: Data-Driven Growth

Section Details
Sales & Order History A filtered view of the Order table showing only the items belonging to that specific seller. It tracks the status of each item (Pending, Shipped, Delivered, or Cancelled).
Listing Visibility (Traffic Tracking) Utilizing the ListingView model, the system logs daily unique views.
Technical Implementation When a customer views a listing, the backend checks: ListingView.objects.get_or_create(listing=listing, date=today). It then increments the view_count.
Performance Visualization This data is served via a specialized API endpoint that returns a time-series JSON object, which the Next.js frontend renders into interactive line and bar charts (using libraries like Recharts or Chart.js).

4. Payouts & Financial Lifecycle

Section Details
Wallet Balance Logic The system tracks Pending Balance (funds from orders not yet delivered) and Available Balance (funds cleared for withdrawal).
Withdrawal Requests Sellers can trigger a withdrawal request. The system validates the Available Balance against the requested amount before creating a Payout record.
Stripe Connect Integration (Future-Ready) While currently handling manual payouts via banking details, the architecture is designed to integrate with Stripe Connect for automated split payments and instant seller onboarding.
Transaction Logs A transparent ledger for the seller, showing a history of every credit (sale) and debit (withdrawal/marketplace fee).

3. System Architecture

The application follows a decoupled Client-Server Architecture:

Frontend (Next.js)

  • Handles UI/UX
  • Client-side routing
  • State management
  • Communicates via REST APIs

Backend (Django REST Framework)

  • Business logic
  • Authentication (JWT)
  • Database ORM using PostGreSQL
  • Third-party integrations

Media Pipeline

  • Images are not stored locally
  • Uploaded to Cloudinary
  • Delivered via CDN

Data Flow & Request Lifecycle

This section describes the high-speed communication between the Next.js Client, the Django API, and the PostgreSQL Database.

The Search Query Journey

Step Title Details
1 The Client Trigger (Frontend) Action: A user enters a keyword (e.g., “Vintage Camera”) into the SearchBar component. Logic: The Next.js frontend uses a “debounce” function to wait until the user stops typing, then initiates a GET request to the backend. Endpoint: GET /api/search/products/?q=vintage+camera
2 The Gateway & Middleware (Backend) Reception: The request hits the Django URL dispatcher (urls.py). Security Check: If the user is logged in, the JWT Middleware validates the token. If they are a guest, the PublicPermission allows the search to proceed. Routing: The request is handed over to the ProductSearchListView.
3 Database Query (ORM Layer) Processing: Django converts the search string into a SQL query. Logic: The system performs a “Fuzzy Search” using icontains. SQL Example: SELECT * FROM api_product WHERE name ILIKE ‘%vintage camera%’; Optimization: The database utilizes indexes on the name and category fields to return results in milliseconds.
4 Serialization (Data Formatting) Transformation: The raw database rows (Objects) are passed to the ProductSerializer. Enhancement: The Serializer calculates the discount_percentage on-the-fly and fetches the full Cloudinary CDN URL for the product images. Output: The data is converted into a clean JSON Array.
5 The Response (Delivery) Transmission: Django sends a 200 OK status code along with the JSON payload. Rendering: The Next.js frontend receives the data, updates the state, and maps the results into ProductCard components using the Virtual DOM for a flicker-free update.

️ Data Integrity Summary

Feature Implementation
Atomic Operations Ensures that partial data is never saved if an error occurs mid-request.
Sanitization Django ORM automatically prevents SQL Injection by parameterizing all search queries.
Image Optimization Instead of sending heavy raw images, the API only sends lightweight URL strings, which the browser loads lazily.
